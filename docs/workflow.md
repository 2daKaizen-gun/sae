# 작업 사이클 — PCTC (Workflow)

> 冴え(Sae)에서 **하나의 작업을 처리하는 표준 루프**.
> **PCTC = Plan → Code → Test → Commit.**
> 헌법 제1조(정확도는 테스트로 증명)·제6조(설명 가능한 커밋)·제9조(한 번에 하나씩)를 실천으로 만든 것.

- **최종 수정:** 2026-07-19

---

## 한눈에

```
  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
  │  1. Plan │ → │  2. Code │ → │  3. Test │ → │ 4. Commit│
  └──────────┘   └──────────┘   └────┬─────┘   └────┬─────┘
        ↑                            │ 실패          │
        └────────────────────────────┘              │ 다음 작업
        └───────────────────────────────────────────┘
```

**한 사이클 = 설명 가능한 한 가지 변경**(제9조 4항). 크면 쪼갠다.

---

## 1. Plan — 계획

**목표:** 무엇을·왜 바꿀지, 성공 기준이 무엇인지 코드 짜기 **전에** 합의한다.

- 작업을 한 문장으로 정의한다: "무엇을, 왜".
- **헌법 체크**: 이 작업이 어떤 조항에 닿는가? 특히 타이밍 경로면 제1조가 최우선.
- **성공 기준**(= Test 단계에서 확인할 것)을 미리 적는다. "정확도는 나중에 테스트"는 없다(제1조 3항).
- 스코프가 MVP 4주 계획(`CONCEPT.md` §7) 밖이면 멈추고 "나중" 목록으로(제7조 3항).
- 결정이 필요한 갈림길이면 **추천안과 함께** 사용자에게 확인. 답에 따라 행동이 바뀔 때만 묻는다(제9조 2항).

**나가는 조건:** 무엇을 만들지, 무엇으로 성공을 판정할지 둘 다 분명하다.

## 2. Code — 구현

**목표:** 계획대로, 주변 코드 스타일로, 최소한으로 만든다.

- **명확함 > 영리함**(제6조 2항). 읽는 사람(면접관·미래의 나)을 위해 쓴다.
- 애플 네이티브 스택 우선, 지금 안 쓰는 추상화 금지(제7조).
- **타이밍 경로에 불확실한 지연을 넣지 않는다**(제1조 1항). 애니메이션·비동기가 측정 루프를 오염시키면 그 방식을 버린다.
- 사용자에게 보이는 문자열은 하드코딩 금지, String Catalog로(제5조 1항).
- 한 사이클에서 여러 개를 동시에 건드리지 않는다. 커지면 Plan으로 돌아가 쪼갠다.

**나가는 조건:** 빌드되고, 계획한 변경만 담겨 있다.

## 3. Test — 검증

**목표:** 성공 기준을 **증명**한다. 됐다고 말하려면 확인이 있어야 한다.

- **핵심(정확도 엔진)에는 테스트를**, 나머지는 실용적으로(제6조 4항). 반응시간 엔진은 예외 없이 테스트로 증명(제1조 3항).
- Plan에서 적은 성공 기준을 하나씩 확인한다.
- 가능하면 실제 동작을 관찰한다(단순 빌드 통과가 아니라 그 흐름을 돌려본다).
- **실패하면 Code로 돌아간다.** 실패를 덮지 않고 그대로 보고한다(제9조 6항).
- 측정의 한계·추정치는 숨기지 않는다(제2조 3항).

**나가는 조건:** 성공 기준이 관찰로 확인됐다. 실패했으면 그 사실이 보고됐다.

## 4. Commit — 기록

**목표:** 변경을 면접에서 설명 가능한 단위로 남긴다.

- `collaboration.md` §7 GitHub 규약을 따른다: **커밋 메시지는 영어**, `type: english summary` + 본문에 **"왜"**.
- 검증 결과를 정직하게 반영한다. 미완이면 미완이라고 커밋 본문/보고에 적는다.
- 설계 결정·트레이드오프가 있었다면 커밋 본문에 남긴다(제8조 3항).
- 커밋·푸시는 **사용자 요청 시**. 자동으로 밀어붙이지 않는다.

**나가는 조건:** 변경이 설명 가능한 커밋으로 남았다. → 다음 작업의 Plan으로.

---

## 사이클을 벗어나는 신호

- Test에서 반복 실패 → Plan의 가정이 틀림. 코드가 아니라 **계획을 다시 본다**.
- 한 사이클이 자꾸 커진다 → **쪼갠다**(제9조 4항).
- 헌법 조항과 부딪힌다 → **헌법이 이긴다**. 조항이 현실과 안 맞으면 코드가 아니라 헌법을 먼저 고친다(개정 절차).
- 단계(Phase)가 끝난다 → `docs/handoff.md`로 인수인계를 남기고 다음 단계로.
