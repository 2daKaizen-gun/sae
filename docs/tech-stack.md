# 기술 스택 (Tech Stack)

> 冴え(Sae)가 쓰는 기술과 **왜 그것을 골랐는지**. 모든 선택은 헌법 조항에 근거를 둔다.
> 원칙은 **애플 네이티브 우선, 외부 의존성 최소, MVP 스코프 사수**(제7조).

- **최종 수정:** 2026-07-19
- **플랫폼:** iOS 네이티브
- **언어:** Swift
- **외부 의존성:** 없음 (기본값). 정말 필요할 때만 추가하고 그 이유를 남긴다(제7조 2항).

---

## 타깃 · 도구

| 항목 | 값 | 근거 |
|------|-----|------|
| 최소 iOS 버전 | **iOS 17.0+** | SwiftData·Swift Charts·Observation이 iOS 17부터. (상향 검토는 아래 열린 결정) |
| UI | **SwiftUI** | 네이티브 우선(제7조). 선언형으로 명확함>영리함(제6조) |
| 상태 관리 | **Observation (`@Observable`)** | iOS 17 네이티브. 외부 상태 라이브러리 불필요 |
| IDE / 빌드 | Xcode | 표준 |
| 테스트 | **Swift Testing** (+ 필요 시 XCTest) | 정확도 엔진을 테스트로 증명(제1조 3항) |

## 레이어별 스택 (조항 근거 포함)

### 1. 정밀 타이밍 — `CADisplayLink` · `mach_absolute_time`
**제품의 심장(제1조).** 자극 표시 → 탭 → 기록의 타이밍 경로.
- `CADisplayLink`로 디스플레이 주사율에 동기화해 자극 표시 시점을 프레임 단위로 안다.
- 고정밀 단조 시계(`mach_absolute_time` / `DispatchTime`)로 반응시간을 잰다. 벽시계(`Date`)는 표시용, 측정용 아님.
- **터치·디스플레이 지연을 측정·보정**하고 그 값을 `PVTSession.calibrationOffsetMs`에 남긴다(제1조 2항).
- 타이밍 루프에 애니메이션·비동기·레이아웃 같은 **불확실한 지연을 넣지 않는다**(제1조 1항).

### 2. 저장 — `SwiftData`
- 온디바이스 로컬 저장(제3조). 스키마는 `docs/data-model.md`.
- 반응시간 **원자료(trial 단위)**를 보존해 사후 검증·설명 가능성 확보(제1·2조).
- 서버 동기화 없음. 필요해지면 제3조를 먼저 고친다.

### 3. 차트 — `Swift Charts`
- 冴え度 추이 그래프. 네이티브(제7조). JPYConverter에서 쓰던 경험 재활용(`CONCEPT.md` §7).
- 숫자를 왜곡 없이 시각화(제2·4조).

### 4. 자율신경 (HRV) — `HealthKit`
- 애플워치 HRV(RMSSD) 읽기. 권한은 **필요한 순간에 이유를 설명하고** 요청(제3조 3항).
- 출처·추정 여부를 데이터에 표시(`HRVReading.source`, `isEstimated`, 제2조).

### 5. 신체 피로 — `CoreMotion`
- 가속도/자이로로 생리적 손떨림 약 10초 측정.

### 6. 국제화 — **String Catalog (`.xcstrings`)**
- 처음부터 일/영/한 3언어, 문자열 하드코딩 금지(제5조 1항).
- 세 언어를 동등하게. 언어별로만 성립하는 「冴え」 말장난은 태그라인으로 보완(제5조 3항).

### 7. 캐릭터 전달 계층 — さえちゃん
- 데이터 계층(冴え 엔진)과 **분리**. 톤은 따뜻하게, 숫자는 정직하게(제4조). 기술이라기보단 UX/카피 설계 계층.

## 명시적으로 **안** 쓰는 것 (제7조)

- **서드파티 패키지 매니저 의존성**(SPM 외부 라이브러리) — 편의를 위해 무게를 늘리지 않는다. 필요해지면 이유를 남기고 도입.
- **추적·광고·분석 SDK** — 넣지 않는다(제3조 2항).
- **백엔드/서버** — 온디바이스 전제(제3조).
- **지금 안 쓰는 추상화·일반화** — 미리 만들지 않는다(제7조 4항).

---

## 열린 결정

- **최소 iOS 버전 상향**(17 → 18 등) 여부 — 최신 API가 필요해지면 재검토. 지금은 17로 넓게.
- **Swift Testing vs XCTest** 비중 — 정확도 엔진은 어느 쪽이든 테스트로 증명이 목표. 기본은 Swift Testing.
- **冴え度 알고리즘** 공식·가중치 — `docs/score-algorithm.md`에 정의(제2조 설명 가능성).
