# 협업 방식 (Collaboration)

> 冴え(Sae)를 **개발자(사용자)와 Claude가 함께** 만드는 방식.
> 헌법 제8·9조(협업 원칙)를 매일 쓸 수 있는 실천 규칙으로 푼 문서다.
> 이 문서와 헌법이 충돌하면 **헌법이 이긴다**(`CONSTITUTION.md`).

- **최종 수정:** 2026-07-19

---

## 1. 역할 분담

| | 개발자 (사용자) | Claude |
|--|----------------|--------|
| **방향** | 무엇을·왜 만들지 결정, 스코프 확정, 최종 심사 | 방향을 코드·문서로 구현 |
| **결정** | 제품·설계 결정의 최종 권한 | 선택지에 **추천안**을 붙여 제시(제9조 1항) |
| **구현** | 리뷰·수정 요청 | 코드 작성, 테스트, 검증, 문서화 |
| **기록** | 큰 결정의 방향 제시 | 결정·이유를 **그 자리에서** 문서·메모리에 남김(제8조 3항) |

핵심: **Claude는 대신 결정하지 않는다. 대신 추측하지도 않는다.** 정해진 건 정해진 대로 따르고(제8조 2항), 애매하면 문서를 다시 본다.

## 2. 소통 규칙

- **대화 언어:** 한국어.
- **산출물 언어:** 코드 주석·문서는 한국어 기준. 사용자에게 보이는 앱 문자열은 String Catalog로 일/영/한(제5조).
- **질문은 답에 따라 행동이 바뀔 때만**(제9조 2항). 그 외에는 합리적 기본값을 택하고 "이렇게 했다"고 알린다.
- **불확실하면 불확실하다고 말한다**(제9조 5항). 확신을 꾸며내지 않는다.
- **실패·건너뜀·미완은 그대로 보고한다**(제9조 6항). 됐고 검증됐으면 담백하게 됐다고 한다.

## 3. 진실의 원천 (우선순위)

헷갈리면 기억이 아니라 이 순서로 다시 본다(제8조 1항):

1. `CONSTITUTION.md` — 최상위 판단 기준 (조항 번호 낮을수록 우선)
2. `CONCEPT.md` — 무엇을 만드는가
3. 코드 · git 히스토리
4. `docs/handoff.md` 최신 인수인계 · 메모리

과거 결정과 **말없이 모순되지 않는다**. 바꿔야 하면 바뀐다고 먼저 밝힌다(제8조 5항).

## 4. 결정을 남기는 곳

| 무엇 | 어디에 |
|------|--------|
| 제품 방향·스코프 변경 | `CONCEPT.md` 갱신 |
| 원칙 자체의 변경 | `CONSTITUTION.md` 개정 절차 |
| 설계 결정·트레이드오프 | 커밋 메시지 + 필요 시 `docs/` |
| 단계 마무리 상태 | `docs/handoff.md` (템플릿 따라) |
| 세션 넘어 살아남을 사실 | 메모리 |

## 5. 시작 리추얼 (세션·페이즈)

**모든 세션이나 페이즈를 시작하기 전에**:

1. `CONSTITUTION.md`, `collaboration.md`, `workflow.md`, `handoff.md` 네 문서를 읽는다.
2. 사용자에게 **읽었음을 알리고**, `handoff.md` 최신 인수인계의 "다음 할 일"을 요약 보고한다.
3. 사용자의 **진행 확인(go)을 받은 뒤** 작업을 시작한다. 확인 전에는 코드·구현을 시작하지 않는다.

- **세션 시작**은 `.claude/settings.json`의 `SessionStart` 훅이 이 리추얼을 자동으로 상기시킨다.
- **페이즈 경계**(예: 1주차→2주차)는 훅이 감지할 수 없으므로 **이 문서를 근거로 직접** 리추얼을 수행한다.
- **페이즈 시작 시 추가로**: 그 페이즈의 GitHub 이슈를 먼저 만들어 올린 뒤 작업을 시작한다(§7 GitHub 규약).

## 6. 작업 리듬

- **한 번에 하나씩, 확인하고 넘어간다**(제9조 4항). 검증 없이 변경을 쌓지 않는다.
- 실제 작업 사이클은 `docs/workflow.md`의 **PCTC(Plan → Code → Test → Commit)**를 따른다.
- 단계(Phase) 경계에서는 `docs/handoff.md` 템플릿으로 인수인계를 남긴다.
- 맥락이 요약·압축된 뒤에도 **하던 일을 이어서 한다**. 조기 마무리·인수인계하듯 굴지 않는다(제8조 4항).

## 7. GitHub 규약

헌법 제6조: 모든 커밋은 면접에서 설명 가능해야 한다. 공개 레포는 포트폴리오의 일부다.

**언어 — 커밋 메시지와 이슈는 무조건 영어.**
일본 취업 포트폴리오라, 공개 기록(커밋·이슈·PR)은 리뷰어·면접관이 읽기 좋게 **영어**로 쓴다.
반면 **내부 문서(`docs/`)와 대화는 한국어**를 유지한다 — 둘은 다른 계층이다.

### 커밋
- **형식:** `type: english one-line summary` + 본문에서 **"왜"를 영어로** 설명.
  `type` ∈ `feat` / `fix` / `docs` / `test` / `refactor` / `chore`.
- **작게, 자주.** 한 커밋 = 설명 가능한 한 가지 변경.
- 관련 이슈가 있으면 본문/푸터에 `Refs #N`, 페이즈 완료 커밋은 `Closes #N`.
- 본문 끝에 공동 작성자 표기:
  ```
  Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
  ```
- 커밋·푸시는 **사용자가 요청할 때** 한다. main에 직접 쌓지 않는 브랜치 전략은 필요해지면 도입.

### 이슈 (Issue)
- **페이즈를 시작하기 전에 그 페이즈의 이슈를 먼저 만들어 올린다**(§5 시작 리추얼과 연결). 무엇을·왜·완료 기준을 이슈에 먼저 적고 나서 코드를 시작한다.
- **영어로** 작성한다.
- **제목:** `Phase N: <goal in english>`.
- **본문(권장 틀):**
  ```markdown
  ## Goal
  <what this phase delivers and why>

  ## Scope (checklist)
  - [ ] <task>

  ## Done when
  - <observable completion criteria — 정확도 엔진이면 테스트 통과 명시(제1조)>

  ## Constitution
  - <관련 조항, 예: Article 1 (measurement accuracy)>
  ```
- 페이즈가 끝나면 이슈를 닫는다(완료 커밋의 `Closes #N` 또는 수동 close).
